# Спецификация Note Turner

Живой документ — единый источник правды по проекту. По нему сверяем результаты
разработки, чтобы не отклоняться от курса. Обновляется вместе с кодом
(см. раздел «Регламент ведения документа»).

Актуально: Фазы 1-3 готовы, Фаза 4 (RAG) в работе — подключён источник Google Drive
с векторным хранилищем.

---

## 1. Обзор и цель продукта

Note Turner — Telegram-бот, корпоративный ассистент компании «Виртуозы».

- Собирает знания из CRM Hollihop (лиды, ученики, платежи) в собственное хранилище.
- Отвечает сотрудникам на вопросы в Telegram, используя языковые модели через OpenRouter.
- Разграничивает доступ: обычные сотрудники и администраторы; финансовые данные — только для админов.

Ценность: единая точка ответов на рабочие вопросы на основе данных CRM,
без ручного поиска по системе.

---

## 2. Заинтересованные стороны и роли

- **Главный администратор** — задаётся в env (`ADMIN_TELEGRAM_ID`), не может быть удалён.
  Его личный чат всегда работает как `assistant`, даже без регистрации.
- **Администраторы** — добавляются существующими админами (хранятся в таблице `admins`).
  Управляют чатами, запускают выгрузку CRM, видят статистику и финансовые данные.
- **Сотрудники** — задают вопросы боту в зарегистрированных чатах.

Поведение по типу чата:

- **Личные сообщения** — отвечает на любой текст (кроме команд).
- **Группа / супергруппа** — отвечает только при @упоминании бота.
- **Канал** — игнорирует.

## 3. Функциональные требования

### 3.1. Бот и доступ

- Точка входа: FastAPI + aiogram, режимы `webhook` (prod) и `polling` (local),
  переключение через `BOT_MODE` ([main.py](../src/noteturner/main.py)).
- Команды:
  - `/start` — приветствие;
  - `/ping` — проверка живости ([bot/handlers/ping.py](../src/noteturner/bot/handlers/ping.py));
  - `/status` — статус сервиса;
  - `/admin` — админ-меню (только админы);
  - `/admins` — список администраторов;
  - `/addadmin <telegram_id>` — добавить админа;
  - `/deladmin <telegram_id>` — удалить админа (кроме главного).
- Разграничение доступа реализуют фильтры и middleware:
  [bot/filters.py](../src/noteturner/bot/filters.py),
  [bot/middlewares/chat_access.py](../src/noteturner/bot/middlewares/chat_access.py),
  [bot/access.py](../src/noteturner/bot/access.py).

### 3.2. Роли чатов

Чат регистрируется администратором и получает роль:

- `assistant` — отвечает на вопросы через OpenRouter.
- `collector` — молча сохраняет сообщения в БД (`collector_messages`).
- не зарегистрирован — в личке просит обратиться к админу, в группе молчит.

### 3.3. Ассистент и LLM

Ответы собирает `Answerer` ([services/llm/](../src/noteturner/services/llm/)):

- Классифицирует вопрос как `simple` или `complex` (ключевые слова + порог длины).
- Выбирает модель по маршрутизации с fallback: при ошибке модели берётся следующая,
  затем общий `fallback`.
- Использует системный промпт «Виртуозы».
- Конфигурация без правки кода:
  [config/routing.yaml](../src/noteturner/config/routing.yaml) — модели и правила классификации;
  [config/prompts.yaml](../src/noteturner/config/prompts.yaml) — системный промпт.
- Каждый запрос логируется в `query_logs`.

Если настроен Google Drive, используется `VectorRetriever`
([services/llm/retriever.py](../src/noteturner/services/llm/retriever.py)): вопрос
эмбеддится, из `doc_chunks` берутся ближайшие фрагменты (с учётом `include_financial`),
и они подмешиваются в ответ с источниками. Иначе — `NullRetriever` (без контекста).

### 3.4. Синхронизация CRM (Hollihop)

- Ручная выгрузка через админ-меню ([services/crm_sync.py](../src/noteturner/services/crm_sync.py)).
- Выгружаемые типы записей: `lead` (`GetLeads`), `student` (`GetStudents`),
  `payment` (`GetPayments`, финансовые).
- Финансовые записи сохраняются, но помечаются флагом `is_financial=True` в `raw_records`.
- Каждая выгрузка трекается записью в `sync_runs` (статус, число записей, ошибки).
- Постранично, ограничения: `PAGE_SIZE = 100`, `MAX_RECORDS_PER_TYPE = 500`.

### 3.5. Источник знаний Google Drive (RAG)

- Доступ к папке через сервисный аккаунт Google ([integrations/gdrive.py](../src/noteturner/integrations/gdrive.py)).
- Ручная выгрузка через админ-меню ([services/drive_sync.py](../src/noteturner/services/drive_sync.py)):
  обход папки → извлечение текста (Google Docs/Sheets/Slides, PDF) → чанкинг →
  эмбеддинги через OpenRouter (`/embeddings`) → сохранение в `doc_chunks` (pgvector).
- Финансовые файлы помечаются эвристикой по имени (`FINANCIAL_KEYWORDS`), в контексте
  доступны только администраторам (`search_chunks(include_financial=...)`).
- Каждая выгрузка трекается записью в `sync_runs` (source `gdrive`).

### 3.6. Здоровье сервиса

- Эндпоинт `GET /health` проверяет PostgreSQL, Telegram, OpenRouter, Hollihop, Google Drive
  ([health/checker.py](../src/noteturner/health/checker.py)).

## 4. Модель данных

Схема в [db/models.py](../src/noteturner/db/models.py), миграции в [alembic/](../alembic/):

- `admins` — администраторы (`telegram_id`, кто добавил, дата).
- `chats` — зарегистрированные чаты и их роль (`assistant` / `collector`).
- `collector_messages` — сообщения из чатов-коллекторов (автор, текст, дата).
- `raw_records` — сырые записи из источников (CRM): тип, внешний id, текстовая
  сериализация `content` (для будущего RAG), сырой `payload`, флаг `is_financial`.
- `sync_runs` — журнал выгрузок (источник, статус, число записей, лог ошибок).
- `doc_chunks` — векторные фрагменты документов из Google Drive (`embedding` pgvector,
  `content`, `title`, `record_type`, `is_financial`) для семантического поиска.
- `query_logs` — журнал вопросов к ассистенту (чат, вопрос, использованная модель).

Требуется расширение PostgreSQL `pgvector` (создаётся миграцией `0003`).

## 5. Дорожная карта по фазам

- **Фаза 1 — Инфраструктура — Готово.**
  FastAPI + aiogram (webhook/polling), health-check, PostgreSQL + SQLAlchemy + Alembic, деплой на Render.
- **Фаза 2 — Сбор данных и роли — Готово.**
  Роли чатов `assistant`/`collector`, сохранение сообщений, ручная выгрузка Hollihop
  в `raw_records`, флаг `is_financial`, трекинг `sync_runs`.
- **Фаза 3 — Мультиадмины и маршрутизация моделей — Готово (текущая).**
  Меню `/admin`, мультиадмины (главный из env + БД), классификация simple/complex,
  маршрутизация OpenRouter с fallback, системные промпты в YAML, лог `query_logs`.
- **Фаза 4 — RAG / контекст из данных — В работе.**
  Реализован источник Google Drive: извлечение текста, эмбеддинги через OpenRouter,
  векторное хранилище `doc_chunks` (pgvector), `VectorRetriever` подмешивает контекст
  в ответы с источниками и фильтрацией финансов для не-админов.
  Осталось: векторизация `raw_records` (CRM) и `collector_messages`, тюнинг поиска.

### Фаза 4b — Отчёты и аналитика (планируется)

- Генерация форматированных отчётов/аналитики (маркетинг и т.д.) поверх векторного хранилища.

### Бэклог (за пределами текущих фаз)

- Автоматическая синхронизация источников по расписанию (сейчас только вручную).
- Векторизация CRM (`raw_records`) и `collector_messages` наравне с Google Drive.
- Расширение списка выгружаемых сущностей Hollihop.

## 6. Критерии приёмки

- **Фаза 1:** сервис поднимается в обоих режимах; `GET /health` возвращает статусы
  всех зависимостей; миграции применяются `alembic upgrade head`.
- **Фаза 2:** админ может зарегистрировать чат с ролью; `collector` сохраняет
  сообщения; ручная выгрузка наполняет `raw_records`, финансовые помечены
  `is_financial`; выгрузка отражена в `sync_runs`.
- **Фаза 3:** `/admin` доступно только админам; админы добавляются/удаляются
  (кроме главного); вопросы классифицируются и уходят на модель по правилам с
  fallback; промпты и модели меняются через YAML без правки кода; вопросы попадают
  в `query_logs`.
- **Фаза 4:** файлы из Google Drive выгружаются и векторизуются в `doc_chunks`;
  релевантный контекст подмешивается в ответ с источниками; финансовые фрагменты
  не попадают в контекст для не-админов.

## 7. Нефункциональные требования

- Python `>=3.12`; зависимости в [pyproject.toml](../pyproject.toml).
- Конфигурация через переменные окружения (перечень — в [README](../README.md#переменные-окружения)).
- Миграции БД — Alembic; на Render применяются автоматически при старте контейнера.
- Деплой — Render по [render.yaml](../render.yaml).
- Тесты — pytest ([tests/](../tests/)); линтер — ruff (line-length 100).

---

## 8. Вне области (Non-goals)

- Веб-интерфейс / админка вне Telegram.
- Редактирование данных CRM из бота (только чтение/выгрузка).
- Голосовые/медиа-ответы, генерация файлов.
- Поддержка источников данных, кроме Hollihop (на текущих фазах).
